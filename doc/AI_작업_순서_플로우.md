# AI 협업 페어 프로그래밍 작업 순서 플로우 (표준화 지침)

본 문서는 AI(Gemini 등)와의 페어 프로그래밍을 통해 웹 애플리케이션 및 플랫폼을 구축할 때, 어떤 절차로 커뮤니케이션하고 어떤 산출물(문서 및 코드)을 남겨야 하는지 정리한 표준 가이드입니다. 
차후 유사한 프로젝트가 진행될 때 이 플로우를 기반으로 AI에게 지시를 내리면 훨씬 구조적이고 누락 없는 결과물을 얻을 수 있습니다.

---

## 🚀 1단계. 비즈니스 문서 단계 (기획 및 요구사항 정의)
**목적:** 프로젝트의 핵심 철학, 기능 명세, 데이터 모델링 등 기초 공사를 수행합니다.
- **작업 내용:**
  - 사용자의 비즈니스 목표 파악 (예: 한일 부부를 위한 보이스 중심의 편안한 소통 공간).
  - 핵심 요구사항 및 페르소나 정의.
  - 마일스톤 및 릴리즈 범위 확정.
- **실제 파일 경로 (산출물):** 
  - `plan/동네꿀팁_고도화_설계플랜.md`
  - `doc/기획서.md` (필요시)
- **추가 개선사항:** 기획 변동 이력을 추적할 수 있도록 `Changelog`나 `Revision History`를 비즈니스 문서 상단에 표기하는 규칙을 추가하면 좋습니다.

---

## 🎨 2단계. 디자인 문서 단계 (UI/UX 및 마크업)
**목적:** Tailwind CSS 등을 활용한 마크업 기초를 잡고, 디자인 토큰(색상, 폰트)을 확정합니다.
- **작업 내용:**
  - 글로벌 스타일링 규칙 확정 (Colors, Fonts 등).
  - 와이어프레임 기반의 HTML 컴포넌트 분리 (`nav.html`, 바텀시트 등).
  - 반응형 분기(`md:`, 모바일 First) 및 감성 연출(글래시즘, 애니메이션).
- **실제 파일 경로 (산출물):** 
  - `components/nav.html`
  - `host.html`, `index.html`, `dongnae.html` 등 각종 뷰 파일.
- **추가 개선사항:** 공통으로 쓰이는 아이콘 체계(`iconify-icon`)나 애니메이션 클래스(`.reveal-slow` 등)를 하나의 CSS 파일(`style.css` 또는 `config.js`)로 완벽히 분리하여 재사용성을 극대화해야 합니다.

---

## 🏗️ 3단계. 인프라 문서 단계
**목적:** 서버, DB, CDN, PWA 등 서비스 구동을 위한 인프라 아키텍처를 설계하고 세팅합니다.
- **작업 내용:**
  - Firebase Firestore 보안 룰 및 데이터 구조 세팅.
  - 서비스 워커(Service Worker) 및 PWA `manifest.json` 설정.
  - 호스팅 및 캐시 서버 설정 (Cloudflare Edge Cache 등).
- **실제 파일 경로 (산출물):** 
  - `sw.js` (서비스워커)
  - `manifest.json`
- **추가 개선사항:** 환경 변수(`.env`) 템플릿과 로컬 디버깅/배포용 스크립트(`package.json` 등) 관리 지침 문서를 추가하면 협업이 더 매끄러워집니다.

---

## 💻 4단계. 개발 문서 단계 (코어 로직 구현)
**목적:** 비즈니스 로직(JS)을 구현하고, 모듈화하여 관리합니다. **범용적으로 사용 가능한 기술과 트러블슈팅 내역은 이 단계에서 반드시 별도로 문서화**합니다.
- **작업 내용:**
  - 음성 인식(`voiceManager.js`), 번역 모듈(`translateManager.js`), 데이터 레이어(`dataService.js`).
  - 외부 API 연동 (네이버 지도, 토스 딥링크 등).
- **실제 파일 경로 (산출물):** 
  - `js/translateManager.js`
  - `js/dongnae.js`
  - `doc/google_translate_automation.md` (범용 기술 문서)
  - `doc/pwa_installation_and_cache_guide.md` (범용 기술 문서)
- **추가 개선사항:** JSDoc 등을 활용해 함수 위쪽에 파라미터와 리턴 값을 명세하는 룰을 적용하면 AI가 코드를 읽고 유지보수하는 속도가 비약적으로 향상됩니다.

---

## 📝 5단계. 개발 -> 로그 문서 단계
**목적:** 매 개발 세션(AI와의 대화)이 끝날 때마다 수행한 작업 내용, 원인, 결과를 기록합니다.
- **작업 내용:**
  - 수정된 파일, 새롭게 생긴 버그, 우회(Workaround) 처리한 로직 등에 대한 타임라인 작성.
  - 데일리 스크럼(Daily Scrum) 형식의 요약 기록.
- **실제 파일 경로 (산출물):** 
  - `log.md`
- **추가 개선사항:** 날짜뿐만 아니라 관련된 **이슈 태그** (예: `[UI/UX]`, `[Bugfix]`, `[Feature]`)를 달아 검색과 추적이 용이하게 관리해야 합니다.

---

## 🏁 6단계. 완료 문서 단계
**목적:** 특정 마일스톤이나 버전(v1.0 등) 배포 시 전체 시스템의 상태를 스냅샷으로 찍어 기록합니다.
- **작업 내용:**
  - 최종 릴리즈 노트 작성.
  - 해결되지 않은 채로 넘어가는 기술 부채(Technical Debt) 기록.
  - 차후 업데이트(2차 고도화) 플랜 인수인계.
- **실제 파일 경로 (산출물):** 
  - `README.md`
  - `plan/v2_업데이트_플랜.md`
- **추가 개선사항:** 프로젝트 코드 베이스(폴더 구조도)와 아키텍처 다이어그램(Mermaid 등 활용)을 첨부하면 새로운 인력이 투입되었을 때나 AI 세션이 리셋되었을 때 문맥을 즉시 복구할 수 있습니다.
