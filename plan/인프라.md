[DB 선택 결정: Firestore 채택 (Firebase SQL Connect 비채택)]

검토일: 2026-04-22
Firebase 프로젝트: aitomo-e108f (cdy3088@gmail.com)

Firebase 콘솔에 현재 Firebase SQL Connect (= Data Connect) 기능이 활성화되어 있으나,
비용 구조 및 마이그레이션 용이성을 검토한 결과 1차 런칭 DB는 Firestore를 채택함.

[ Firebase SQL Connect (Data Connect + Cloud SQL PostgreSQL) 분석 ]

특징:
- Google Cloud의 Cloud SQL for PostgreSQL을 Firebase 콘솔에서 제어하는 래퍼(Wrapper) 서비스
- GraphQL 스키마 기반으로 타입 세이프 SDK 자동 생성 (Flutter/Web/Android/iOS용)
- 클라이언트에서 직접 SQL 쿼리가 아닌 Firebase가 생성한 SDK를 통해 접근

비용 구조 (치명적 문제):
- Cloud SQL 인스턴스 자체에 고정 과금 발생: 최소 db-f1-micro(1vCPU, 0.6GB) 기준 월 약 $7~$10 USD 이상
- 연결 수 기반 과금 + 스토리지 과금 + I/O 과금 별도 청구
- Firebase 무료 플랜(Spark)에서는 SQL Connect 사용 불가. Blaze(종량제) 플랜 필수
- 트래픽이 없어도 인스턴스가 살아있는 한 고정 비용 발생 → 0원 운영 목표에 정면 위배
- No-cost trial 존재하지만 프리뷰 기간 한정이며 서비스 후 즉시 과금 시작

마이그레이션 관련:
- PostgreSQL 기반이므로 차후 Oracle Cloud 무료 VM 등에 셀프 호스팅 PostgreSQL로 전환 시
  데이터 덤프(pg_dump)만으로 이전 가능 → 데이터 이식성은 우수
- 단, Firebase SQL Connect SDK를 걷어내고 직접 REST API 호출로 교체하는 클라이언트 코드 수정 필요
  (현재 DataService.js 추상화 레이어로 전환 비용은 최소화 가능)

결론: 초기 운영 비용 발생 문제로 1차 런칭에 부적합

[ Firestore (NoSQL) 분석 ]

비용 구조 (적합):
- 무료 한도(Spark 플랜): 읽기 50,000건/일, 쓰기 20,000건/일, 삭제 20,000건/일, 1GB 저장
- 초기 커뮤니티 서비스 규모에서는 사실상 무료로 운영 가능
- 과금은 실제 요청 횟수 기반 → 트래픽 없으면 비용 없음
- 오프라인 캐시(Persistence) + Pagination 조합으로 읽기 발생 건수 극도로 억제 가능

마이그레이션 관련:
- NoSQL(문서형)이므로 차후 PostgreSQL로 전환 시 스키마 설계 재작업 필요
- 단, 현재 DataService.js 추상화 레이어로 프론트엔드 코드 수정 최소화 가능
- Firestore 데이터는 Export(JSON/CSV) → 정제 → INSERT 방식으로 이전 가능

결론: 초기 0원 운영 목표 달성 가능, 1차 런칭 DB로 채택

[이미지 업로드 플로우 및 규칙]

1. 클라이언트: 
   - 업로드 전 브라우저에서 이미지 리사이징(최대 1200px) 및 WebP 포맷 변환 (최대 300KB 이하로 압축)
2. CF Worker (Presigned URL 발급):
   - Firebase Auth 토큰 검증 후 URL 발급 (미인증자 차단)
   - 서명 생성 시 `Content-Length-Range` 조건을 추가하여 300KB 초과 파일 업로드 원천 차단
   - 단일 IP/유저당 URL 발급 횟수 제한 (Rate Limiting)
3. R2 스토리지:
   - 클라이언트에서 R2로 직접 PUT 업로드
